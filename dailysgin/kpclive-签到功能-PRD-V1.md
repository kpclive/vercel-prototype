## 1. 背景与目标

### 1.1 背景

kpclive 当前缺少能够稳定拉动用户每日打开 App 的轻量化机制。签到作为一种成本低、用户认知成熟的功能形态，可用于培养用户的每日访问习惯。

### 1.2 业务目标

提升日活（DAU），培养用户每天打开 App 的习惯。

### 1.3 目标的预期边界

V1 入口仅放置在个人中心二级页面，不做任何首页层面的曝光提示（banner、气泡、角标等）。这会显著降低签到功能的覆盖率，业务方接受该折损，V1 的核心目的是先验证功能形态与积分激励规则的合理性。

### 1.4 非目标

- 不在 V1 验证奖励体系本身的复杂玩法
- 不在 V1 解决跨端通知 / 提醒回访问题
- 不在 V1 引入社交比拼或徽章体系

## 2. 范围说明

### 2.1 V1 在做

- 个人中心二级签到入口
- 每日签到一次的核心交互
- streak（连续签到天数）展示与计算
- 当日 / 次日积分预告
- 与积分系统解耦的事件触发
- 后台可配置的积分规则数组

### 2.2 V1 不做（V2+ 待办池）

数据埋点、首页 banner 与任何曝光提示、累计签到天数展示、补签卡、积分商城、签到日历可视化、7 日签到任务、排行榜、好友 PK、签到提醒推送、分享海报、等级 / 徽章、节日翻倍、首次签到额外奖励。

### 2.3 模块依赖

- **积分系统**（外部模块）：订阅签到事件、按规则发放积分、提供"预告积分"计算能力
- **运营平台**（外部模块）：提供积分规则数组的配置入口与权限管控

## 3. 用户故事

1. 作为新用户，我进入个人中心后能看到签到入口，点击进入即可完成首次签到并获得积分。
2. 作为已签到用户，我能直观看到自己的连续签到天数和明天可获得的积分，从而有动力第二天再来。
3. 作为中断后回归的用户，我看到 streak 已经归零但累计积分不受影响，签到反馈给我一句安抚文案，让我愿意重新开始。
4. 作为运营，我可以在后台调整每天对应的积分数值与封顶天数，调整后立刻全量生效，不影响历史数据。

## 4. 功能详细说明

### 4.1 入口

- 位置：仅个人中心二级入口
- 不做：首页 banner、任何强制弹窗、首次气泡引导、红点角标
- 入口形态：列表项形式，左侧为"签到"标题与简短副标题，右侧根据当日是否已签到展示不同状态

### 4.2 签到行为

- 频率：每个自然日一次
- 重置：按服务器时区（北京时间）0 点重置
- 幂等：服务端基于 `user_id + date` 唯一索引保证幂等，重复请求返回成功态及当日实际所得积分，不重复发放
- 即时性：积分立刻到账，无延迟规则

### 4.3 展示信息

签到页主要展示两个数字：

- **今日签到状态**：已签 / 未签
- **连续签到天数（streak）**：核心激励指标

**故意不展示**：累计签到天数（在 V1 中没有业务用途，避免认知负担）。

### 4.4 按钮文案规则

| 状态 | 文案 | 数据来源 |
|---|---|---|
| 未签到 | 签到 +N 积分 | N = 用户今日签到将获得的积分 |
| 已签到 | 明天签到可得 +M 积分 | M = 用户次日签到将获得的积分 |

N 和 M 均由积分系统计算后通过接口返回，签到模块不本地维护积分数值。

### 4.5 签到反馈文案

| 场景 | 文案 |
|---|---|
| 普通签到（正常连续 / 用户首次签到） | 签到成功 +N 积分 |
| 中断后首次签到（用户存在历史签到记录，且上一次签到日期早于昨天） | 签到成功 +N 积分，重新出发！ |

**重要边界**："重新出发"文案仅对存在历史签到记录且 streak 已中断的用户生效；首次签到的新用户使用普通文案。

### 4.6 streak 计算规则

- **累加**：用户在 D-1 已签到，D 完成签到时 streak += 1
- **归零**：用户上一次签到日期早于 D-1（即至少漏签一整天），且 D 首次签到时 streak 重置为 1
- **展示口径（懒计算）**：客户端 / 服务端在读取 streak 时，判断"上次签到日期是不是今天或昨天"：
  - 是 → 显示存储的 streak 值
  - 否 → 显示 0
- 不需要定时任务主动重置 streak

### 4.7 积分规则

- 配置形式：后台维护一个**整数数组**，索引 `i` 对应"完成连续第 `i+1` 天签到"应得的积分
- 默认配置：`[5, 6, 7, 8, 9, 10, 15]`
- 数组最后一个值作为封顶后每天的积分；数组长度即封顶天数
- streak 中断归零，但**已发放积分不动**
- 运营修改后**立刻全量生效**，不追溯历史

#### 配置校验规则（运营平台需实现）

- 数组不能为空
- 每个元素必须为非负整数
- 数组长度上限 30
- 修改操作需要二次确认与操作日志

### 4.8 连续签到奖励卡片展示规则

签到主页的奖励卡片用于让用户**直观看到积分规则与当前进度**，定位为"规则展示为主，进度感知为辅"。

#### 4.8.1 固定 7 格展示

V1 卡片**固定展示 7 个格子**（第 1 天 - 第 7 天），不随运营配置数组长度变化：

- 当运营配置长度 < 7：超出数组长度的格子使用数组**最后一个值**（封顶值）填充
- 当运营配置长度 > 7：只展示前 7 个，第 8 个及之后的奖励在卡片上**不可见**

> 设计权衡：保持 UI 稳定、研发成本最低。当运营配置长度 > 7 时，用户依然能通过按钮副标题与签到反馈感知到自己当前的积分；封顶提示文案兜底（见 4.8.3）。横滚展示完整数组放入 V2 待办池。

#### 4.8.2 三种状态下的格子样式

| 用户状态 | 格子样式规则 |
|---|---|
| 未签到（含中断重启 streak=0、新用户） | 第 (streak + 1) 格标"今天"高亮；其余格子默认态 |
| 已签到 | 第 1 至第 streak 格标"已签 done"态；其余格子默认态 |
| 已封顶（streak ≥ 7） | 所有 7 格均标"已签 done"态；不再标"今天" |

**故意不做**："明天"标识。明天的积分预告已经在签到按钮副标题中展示（"明天签到可得 +M 积分"），卡片再做一遍属于信息冗余。

#### 4.8.3 封顶提示文案

当 streak ≥ 7（即用户已达或超过封顶天数）时，在 7 格下方追加一行文案：

```
🎉 已连续签到 X 天，享受最高 Y 积分/天
```

其中 X = 当前 streak，Y = 积分配置数组的最后一个值。该文案的作用是：

- 让封顶用户看到自己的实际连续天数，避免"7 格全 done"带来的进度感缺失
- 明确告知用户已享受最高奖励

#### 4.8.4 中断重启时

用户中断后回归（streak 懒计算 = 0），卡片**完全重置**：

- 第 1 格标"今天"高亮
- 不展示任何历史进度痕迹（不显示"曾连续 X 天"）

> 设计意图：让用户感觉是新的开始；"重新出发"的情绪安抚由签到成功文案承担，卡片不再强化挫败感。

## 5. 接口契约

### 5.1 查询签到状态

`GET /api/checkin/status`

**响应字段**：

| 字段 | 类型 | 说明 |
|---|---|---|
| `checked_in_today` | bool | 今日是否已签到 |
| `streak` | int | 当前连续签到天数（懒计算后的值） |
| `today_preview_points` | int | 今日签到将获得的积分（`checked_in_today=false` 时使用） |
| `tomorrow_preview_points` | int | 明日签到将获得的积分（`checked_in_today=true` 时使用） |

### 5.2 提交签到

`POST /api/checkin`

**响应字段**：

| 字段 | 类型 | 说明 |
|---|---|---|
| `success` | bool | 是否签到成功（含幂等成功） |
| `streak` | int | 签到后的新 streak |
| `points_awarded` | int | 本次获得的积分 |
| `tomorrow_preview_points` | int | 明日签到将获得的积分 |
| `is_restart` | bool | 是否触发"重新出发"文案（中断后首次签到为 true） |

### 5.3 事件契约（签到模块 → 积分系统）

事件名：`user_checked_in`

**事件字段**：

| 字段 | 类型 | 说明 |
|---|---|---|
| `user_id` | string | 用户 ID |
| `date` | date | 签到日期（YYYY-MM-DD） |
| `streak_day` | int | **签到完成后的新 streak 值**（积分系统按此值查积分数组） |

## 6. 数据模型

### 6.1 签到记录表 `user_checkin`

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | bigint | 主键 |
| `user_id` | string | 用户 ID |
| `date` | DATE | 签到日期（YYYY-MM-DD） |
| `streak_day` | int | 签到完成时的 streak 值 |
| `created_at` | timestamp | 创建时间 |

**索引**：`UNIQUE(user_id, date)` —— 用于幂等保证；`INDEX(user_id, date DESC)` —— 用于快速查询最近签到记录。

> 注：`date` 字段采用 `DATE` 类型而非 `timestamp`，避免同一天多次签到通过时间戳差异绕过唯一约束。

### 6.2 streak 存储

V1 不单独建立 streak 表，直接通过查询签到记录表中"用户最近一次签到记录的 streak_day"获取存储值，结合懒计算规则在读取时判断是否归零。

## 7. 交互与异常场景

### 7.1 正常流程

1. 用户打开个人中心 → 调用 `/api/checkin/status`
2. 渲染签到入口：根据 `checked_in_today` 决定显示"签到 +N 积分"或"明天签到可得 +M 积分"
3. 用户点击入口进入签到页 → 展示 streak 与按钮
4. 用户点击签到按钮 → 调用 `/api/checkin`
5. 收到响应：弹出反馈文案（普通 / 重新出发），按钮变灰显示"已签到"，刷新 streak 数字与次日预告文案

### 7.2 异常场景

| 场景 | 处理方式 |
|---|---|
| 网络异常 / 请求失败 | 前端展示"网络异常，请稍后再试"，不更新 UI 状态 |
| 弱网下重复点击 | 服务端识别幂等返回成功态，前端按成功路径展示；用户最多看到一次"今日已签到"提示 |
| 用户跨设备同日重复签到 | 同样走幂等路径，第二台设备拉取状态时直接展示已签到 |
| 服务器跨天瞬间签到 | 23:59 与 00:00 各算一次合法签到，streak 累加 +2，属预期行为 |
| 运营配置在用户操作中途修改 | 不追溯：用户当次签到按其请求落库瞬间的配置发放；下一次签到按新配置 |

## 8. 开发任务拆分建议

### 签到模块（本期）

- 签到记录表与唯一索引建设
- `GET /api/checkin/status` 实现（含 streak 懒计算）
- `POST /api/checkin` 实现（含幂等处理 + is_restart 判断）
- 发送 `user_checked_in` 事件
- 个人中心入口接入
- 签到页 UI 实现（按钮态、文案、反馈、动效）

### 积分系统（依赖方）

- 订阅 `user_checked_in` 事件
- 根据 streak_day 查规则数组发放积分
- 提供 `today_preview_points` / `tomorrow_preview_points` 计算能力（供签到模块聚合返回）
- 接收并维护积分规则数组配置

### 运营平台（依赖方）

- 积分规则数组配置入口
- 配置校验规则（非空、非负整数、长度 ≤ 30）
- 二次确认与操作日志

## 9. 已知遗留与风险

| 项 | 说明 | 决策 |
|---|---|---|
| 数据埋点 | 没有埋点意味着 V1 上线后无法量化验证"提升 DAU"的业务目标 | 业务方决定 V1 不做，V2 补 |
| 入口曝光 | 仅个人中心入口，覆盖率受限 | 业务方接受折损 |
| 积分价值锚点 | 5/6/.../15 的数值需要与积分系统的"1 积分价值"对齐 | 上线前由产品与积分系统拉通 |
| 安抚文案的强度 | V1 采用纯文案安抚，不展示累计积分数字 | 选择 B 方案，降低跨系统耦合 |

## 10. V2+ 待办池（按优先级排序）

1. **数据埋点**：补齐签到入口曝光、按钮点击、签到成功事件，监控签到 UV/DAU、留存差、streak 分布
2. **首页 banner 入口**：根据 V1 数据决定是否打开，作为提升覆盖率的第一杠杆
3. **签到提醒推送**：对昨天签到、今天未签的用户推送提醒
4. **签到日历可视化**：展示一个月的签到分布
5. **累计签到天数展示**：作为长期老用户的精神激励
6. **补签卡**：消耗道具或积分续 streak
7. **节日 / 周末翻倍**：运营活动维度
8. **7 日签到任务奖励**：与签到系统联动的任务玩法
9. **首次签到额外奖励**：新用户激励
10. **排行榜 / 好友 PK**：社交化玩法
11. **分享海报 / 等级徽章**：传播与长期激励
12. **奖励卡片横滚展示**：当运营配置积分数组长度 > 7 时，卡片支持横向滚动展示完整数组，并自动定位到"今天"附近
