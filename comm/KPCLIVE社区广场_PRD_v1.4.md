## 1. 背景与目标

### 1.1 产品背景

KPCLIVE 是公司围绕自有德州扑克赛事 IP "**KPC LIVE**" 打造的社区型产品，包含 App 与 PC 官网两端。所有业务围绕赛事 IP 展开，核心业务方向包括：

- 赛事活动与赛事资讯
- 德州扑克社区运营（图文 UGC）
- 比赛视频观看与直播（**独立模块**，本期不在本 PRD 范围内）

> **业务红线**：本产品**绝对不涉及**任何形式的在线棋牌游戏。社区广场仅承担**图文内容浏览、社交互动、资讯传播**职能，不提供任何形式的牌局对战、虚拟筹码兑换、博彩或类博彩行为。

### 1.2 模块定位

**社区广场**是 KPCLIVE App 内的核心**图文内容**消费场景，作为信息流主入口，承担以下产品职能：

1. **赛事曝光场**：作为 KPC LIVE 赛事 IP 内容（战报、赛程、图文精彩瞬间）的核心分发渠道
2. **社区氛围场**：聚合德扑爱好者的 UGC 图文内容，建立社区粘性与归属感
3. **创作者激励场**：为选手、KOL、内容创作者提供曝光与涨粉路径

### 1.3 目标与衡量指标

| 目标 | 一级指标 | 二级指标 |
| :-- | :-- | :-- |
| 提升用户消费时长 | 日均人均广场停留时长 ≥ 8 分钟 | 单 Session 浏览动态数 |
| 提升内容生产活跃 | 日均 UGC 发布量 / DAU ≥ 3% | 发布转化率、发布完成率 |
| 提升社交互动深度 | 互动率（点赞+评论+转发+收藏）≥ 12% | 评论率、收藏率 |
| 强化赛事 IP 心智 | 赛事关联动态曝光占比 ≥ 25% | 赛事标签点击率、关联跳转率 |
| 保障内容合规 | 违规内容线上漏出率 ≤ 0.05% | 机审通过率 / 召回率 / 误杀率 |

### 1.4 本期范围与不涉及

**本期 PRD 覆盖**：

- 社区广场 feed 流主页
- 动态详情页
- 发布动态页（**仅图文**）
- 用户个人主页
- 内容机审接入（腾讯云 TMS + IMS）
- **Feed 流与评论排序规则（v1.2 新增）**
- **消息中心 / 站内互动消息（v1.3 新增，v1.4 整章重构）**
- **搜索功能（v1.4 新增）**
- **关注列表 / 粉丝列表 / 拉黑名单（v1.4 新增）**

**本期 PRD 不涉及**（独立 PRD）：

- 打卡功能（另见《打卡功能 PRD》）
- **视频动态、短视频、手牌回放**（另见《视频模块 PRD》）
- **赛事直播、Live 直播间**（另见《赛事直播间 PRD》）
- **私信、即时通讯**（另见《私信功能 PRD》）
- **系统级推送（APNs / FCM / TPNS / 极光等）——本期完全不接入**

> **重要措辞澄清（v1.4）**：上一行所指"系统通知/系统推送"，特指"端外/端内**弹窗 banner**" 与 "APNs / FCM / TPNS / 极光等**系统级推送通道**"。**不包括**站内消息中心承载的"内容性消息"（如赛事提醒、官方公告、审核反馈等）—— 站内消息中心是 in-app pull-only 形态，与上述推送通道无关，可正常承载内容性消息。

- 系统通知 Tab（v1.3 曾考虑独立 Tab，v1.4 确认不做；审核反馈不进消息中心，由动态详情页顶部黄/红条提示）
- 选手数据、赛事报名、积分排行
- 用户"我的"Tab 与"设置"页面（含通知偏好设置等）
- 获赞列表页（v1.4 确认不做——获赞为聚合统计数字，不可点击展开；用户想查具体获赞动态在个人主页"动态"Tab 自行翻阅）
- 不感兴趣反馈面板 / 举报分类原因选择 / 处罚申诉通道
- 新用户引导流程
- 金 V / 蓝 V 用户自助认证申请（本期由运营后台手动开通）
- PC 官网（本 PRD 描述以 App 为主；PC 官网设计另见后续独立 PRD）
- 关注/粉丝列表对**他人**可见（v1.4 仅本人可见自己的；他人主页上数字保留但不可点击，规则后续版本再定）

> 本期产品形态：**纯图文社区**。所有视频、直播、私信能力在本期完全关闭入口，待对应模块上线后再开放。

## 2. 目标用户与典型场景

### 2.1 用户画像

| 用户分群 | 占比预估 | 核心诉求 | 内容偏好 |
| :-- | :-- | :-- | :-- |
| 德扑爱好者/玩家 | 35% | 学习策略、看高手手牌图解、找牌局信息 | 手牌截图分析、GTO 策略图文 |
| KPC 赛事粉丝/观众 | 30% | 关注赛事进展、追选手、看战报 | 官方赛事动态、选手个人动态 |
| 泛德扑社区用户 | 25% | 兴趣社交、了解德扑文化、消遣 | 生活化 UGC、轻松向图文 |
| 选手 / KOL / 创作者 | 10% | 涨粉、内容曝光、商业价值变现 | 自身图文内容发布、互动管理 |

### 2.2 典型场景

**场景 A：赛事粉丝看战报**

> 阿乐是 KPC 长期粉丝，下班路上打开 App，第一眼看到广场首页 Fang 老师发布的"Paulson 世界杯第 2 轮战报"，配图是赛事现场决赛桌图片。她点赞、收藏，并点击进入详情页查看完整文字战报，关注 Fang 老师以便后续接收推送。

**场景 B：玩家分享手牌**

> 老王刚打完一场线下比赛，遇到一个有趣的全下场面。他打开 App 点击发布，上传一张手牌截图，关联 #德扑入门 话题和 "济州岛站" 赛事标签，发布动态。第二天收到 20 条评论和 8 个收藏，老王感到被认可。

**场景 C：新用户氛围探索**

> 小张刚下载 App，进入广场，被混合呈现的赛事图文、UGC 九宫格吸引，连续下滑浏览 15 分钟。他对一位选手的旅拍内容感兴趣，关注了对方主页。

**场景 D：KOL 涨粉**

> 熊达是一位生活化 KOL（济州岛/韩国），定期发布旅拍九宫格。新用户在广场看到她的内容，点击右上角"关注"按钮直接关注。

## 4. 通用规则

### 4.1 登录态规则

| 行为 | 未登录 | 已登录 |
| :-- | :-- | :-- |
| 浏览 Feed 流 | ✅ 可浏览 | ✅ 可浏览 |
| 点击动态详情 | ✅ 可查看 | ✅ 可查看 |
| 点赞 / 评论 / 转发 / 收藏 | ❌ 触发登录引导弹窗 | ✅ 正常 |
| 关注用户 | ❌ 触发登录引导弹窗 | ✅ 正常 |
| 发布动态 | ❌ 触发登录引导弹窗 | ✅ 正常 |

> 登录引导弹窗文案："登录后即可参与互动，加入 KPC 社区"，主按钮"立即登录"，次按钮"再看看"。

### 4.2 内容审核规则（机审 + 人审）

#### 4.2.1 审核对象与策略

| 审核对象 | 审核时机 | 审核策略 | 命中处理 |
| :-- | :-- | :-- | :-- |
| 动态正文文本 | 发布提交时（同步） | TMS 文本审核 | 见 4.2.4 |
| 动态配图 | 图片上传完成时（同步） | IMS 图片审核 | 见 4.2.4 |
| 评论文本 | 评论提交时（同步） | TMS 文本审核 | 同动态正文 |
| 评论图片 | 评论图片上传时（同步） | IMS 图片审核 | 同动态配图 |
| 用户昵称 | 注册 / 修改时（同步） | TMS 文本审核 | 拦截 + 提示重新输入 |
| 用户简介 | 修改时（同步） | TMS 文本审核 | 拦截 + 提示重新输入 |
| 用户头像 | 上传时（同步） | IMS 图片审核 | 拦截 + 提示重新上传 |

#### 4.2.2 接入产品（腾讯云）

| 产品 | 用途 | API | 接入方式 |
| :-- | :-- | :-- | :-- |
| **文本内容安全 TMS** | 所有文本类内容的实时审核 | `TextModeration` | 服务端通过 HTTPS 调用 |
| **图片内容安全 IMS** | 所有图片类内容的实时审核 | `ImageModeration` | 服务端通过 HTTPS 调用 |
| **数据万象 CI** | 与 COS 存储桶联动，二次兜底审核 | 自动审核 | 在 COS 控制台配置规则 |

> 上述产品同属腾讯云"天御内容安全（T-Sec）"产品矩阵。统一计费、统一控制台、统一策略管理。

#### 4.2.3 审核风险类别（必须命中拦截的项目）

腾讯云 TMS / IMS 返回的 `Label` 字段，本产品必须配置拦截以下类别：

| Label | 中文名 | 命中行为 |
| :-- | :-- | :-- |
| `Porn` | 涉黄 / 色情 | 直接 Block |
| `Polity` | 涉政敏感 | 直接 Block |
| `Terror` | 涉暴恐 | 直接 Block |
| `Illegal` | 违法违规 | 直接 Block |
| **`Gamble`** | **涉赌博 / 棋牌** | **直接 Block + 账号风险标记**（业务红线，强制最严策略） |
| `Abuse` | 谩骂 / 攻击 | Block |
| `Ad` | 广告 / 引流 | Review（人工复审） |
| `Spam` | 灌水 / 营销 | Review |
| `Normal` | 正常 | Pass |

#### 4.2.4 审核结果三级处理

腾讯云 API 返回 `Suggestion` 字段，本产品按以下策略处理：

| Suggestion | 含义 | 处理方式 | 用户感知 |
| :-- | :-- | :-- | :-- |
| `Pass` | 通过 | 内容正常发布、对所有人可见 | 无感知 |
| `Review` | 疑似 | 内容**先发后审**，对作者本人可见（带"审核中"标识），对其他用户不可见，进入人工复审队列（30 分钟内反馈） | "审核中"标签 |
| `Block` | 违规 | 内容**直接拦截**，不发布，向作者展示具体拦截原因 | 红色提示弹窗 |

#### 4.2.5 机审流程（时序）

```
用户                  客户端                服务端                腾讯云
 │                     │                     │                    │
 │── 点击发布 ────────→│                     │                    │
 │                     │── 上传图片到 COS ──→│                    │
 │                     │                     │── IMS 审图 ───────→│
 │                     │                     │←── Pass/Review/Block│
 │                     │                     │                    │
 │                     │── 提交动态请求 ────→│                    │
 │                     │                     │── TMS 审文本 ─────→│
 │                     │                     │←── Pass/Review/Block│
 │                     │                     │                    │
 │                     │                     │── 综合裁决 ────────│
 │                     │←── 结果返回 ────────│                    │
 │←── Toast 提示 ──────│                     │                    │
```

> **综合裁决规则**：文本 + 图片任一为 `Block` → 整条 `Block`；任一为 `Review` 且无 `Block` → 整条 `Review`；全部 `Pass` → 整条 `Pass`。

#### 4.2.6 兜底机制

1. **腾讯云 API 调用超时**（> 3 秒）：默认放行进入 `Review` 状态 + 进入人工复审队列，避免影响用户体验
2. **腾讯云 API 报错**：同上
3. **二次兜底**：用户内容存入 COS 时，数据万象 CI 自动触发二次审核（异步），命中违规则后台告警 + 自动下线
4. **用户举报路径**：每条动态/评论的"更多 → 举报"，触达人工审核

#### 4.2.7 棋牌博彩内容特殊处理（业务红线）

- 任何命中 `Gamble` 标签的内容：
  1. 内容直接 Block
  2. 用户账号打上"涉博彩风险"标记，第 1 次警告，第 2 次冻结发布权限 7 天，第 3 次永久封号
  3. 后台同步告警邮件至合规组
- 关键词黑名单（与腾讯云策略并行）：本地维护补充词库，覆盖德扑特定场景的擦边词

### 4.3 内容时间显示规则

| 距当前时间 | 显示形式 |

## 5. 排序规则（v1.2 新增）

> 本章为本期新增内容。设计原则：**无算法团队也能落地**——所有计算可用纯 SQL / 缓存完成，无需任何 ML 模型。

### 5.1 Feed 流排序

#### 5.1.1 基础热度公式（所有动态统一）

```
热度分 = (点赞数 × 1 + 评论数 × 3 + 转发数 × 5 + 收藏数 × 2) ÷ (发布小时数 + 2)
```

**字段说明**：

| 变量 | 含义 | 数据来源 |
| :-- | :-- | :-- |
| 点赞数 | `InteractionStats.like_count` | 实时计数 |
| 评论数 | `InteractionStats.comment_count` | 实时计数 |
| 转发数 | `InteractionStats.repost_count` | 实时计数 |
| 收藏数 | `InteractionStats.favorite_count` | 实时计数 |
| 发布小时数 | `(now - created_at)` 转换为小时（**向下取整**，刚发布按 0 小时计） | 计算字段（与附录 D SQL `TIMESTAMPDIFF(HOUR,...)` 行为一致） |

**设计要点**：

1. 互动权重「转发 5 > 评论 3 > 收藏 2 > 点赞 1」——重互动权重高，转发因扩散性最高
2. 时间分母 `+2`：避免刚发布内容除以 0 / 异常爆分
3. 整体计算复杂度为 O(N)，单条 SQL 即可批量算出，无需任何机器学习模型

**算例**（直观感受）：

| 帖子 | 发布距今 | 点赞 | 评论 | 转发 | 收藏 | 热度分 |
| :-- | :-- | :-- | :-- | :-- | :-- | :-- |
| A | 1 小时 | 100 | 10 | 0 | 5 | (100+30+0+10)/(1+2) = **46.7** |
| B | 6 小时 | 500 | 50 | 10 | 30 | (500+150+50+60)/(6+2) = **95.0** |
| C | 24 小时 | 2000 | 200 | 50 | 100 | (2000+600+250+200)/(24+2) = **119.2** |

C 帖虽然爆款，但接近 24 小时后会很快被新内容追上 → 实现"老内容自然沉底"。

#### 5.1.2 加权倍数（叠加在基础分上）

最终排序分 = 热度分 × ∏(命中的倍数因子)

| 加权场景 | 倍数 | 适用条件 |
| :-- | :-- | :-- |
| 蓝 V 官方账号发布 | × **2.0** | `verify_type = BLUE_V` |
| 金 V 选手发布 | × **1.3** | `verify_type = GOLD_V` |
| 关注用户发布 | × **1.5** | 仅对已登录用户生效；动态作者 ∈ 当前用户关注列表 |
| 关联进行中赛事 | × **1.5** | `MatchTag.status = ONGOING` |
| 运营手动置顶 | **强制置顶** | 不参与热度计算，强制排前 3 位（同时最多 3 条） |

**多倍数叠加示例**：蓝 V 官方 + 关联进行中赛事 = 2.0 × 1.5 = 3.0 倍。

> 运营置顶位由后台「内容管理」配置，详见《KPCLIVE 运营后台 PRD》§3.1。

#### 5.1.3 防刷与多样性规则

| 规则 | 说明 |
| :-- | :-- |
| 同作者频次限制 | Feed 流任一连续 10 条内，同一 `author_id` 最多出现 2 条（避免刷屏） |
| 已浏览过滤 | 用户已完整曝光（≥ 3 秒可见）过的动态，**48 小时内**不再重复推送给该用户 |
| 新人保护 | 注册 < 7 天的新用户发布的**首条**动态，强制给予 100 次保底曝光，避免冷启动死循环 |
| 新发布初始分 | 新动态在还没有任何互动数据时，赋予初始热度分 **10**，避免永远排不到前面 |

#### 5.1.4 时效与计算时机

| 项 | 规则 |
| :-- | :-- |
| 热度分重算频率 | **每 5 分钟批量重算一次**（定时任务，覆盖近 7 天内动态） |
| 内容生命周期 | 发布超过 **7 天**的动态不再参与热度排序，自动沉底（即使是爆款也不例外，沉底后仅在个人主页可见） |
| 用户拉取 Feed 时机 | 按"热度分 × 加权倍数 × 防刷规则"倒序取数；置顶位强制排在最前 3 条 |

#### 5.1.5 异常兜底

| 异常场景 | 兜底策略 |
| :-- | :-- |
| 热度计算服务故障 | 降级为按 `created_at` 时间倒序展示 |
| Feed 候选池为空 / 全部已读 | 降级为按时间倒序展示官方账号（蓝 V）内容 |
| 用户长期无互动 | 不做特殊推荐，按全站热度通用规则 |

#### 5.1.6 排序优先级（最终输出顺序）

```
最终 Feed 顺序 =
   ① 运营置顶位（最多 3 条，按后台配置顺序）
        ↓
   ② 候选池按 "热度分 × 加权倍数" 倒序
        ↓
   ③ 应用防刷规则（同作者频次、已浏览过滤）
        ↓
   ④ 应用新人保护（保留至少 1 个曝光位）
        ↓
   ⑤ 兜底（候选不足时补充时间倒序的官方内容）
```

### 5.2 评论排序

#### 5.2.1 一级评论

**默认：热度排序**（用户可手动切换为"时间排序"）。

```
评论热度分 = (点赞数 + 1) ÷ (发布小时数 + 2)
```

- 「+1」保证无点赞评论也有非零起步分（`1 / (0+2) = 0.5`）
- 一旦有人点赞，立刻往前排
- 公式与 Feed 流相比简化（不区分转发/收藏，因为评论无此交互）

**时间排序**：按 `created_at` 倒序，最新评论在前。

**作者优先（不论排序模式）**：
- 动态作者本人参与的评论（无论回复谁），**永远排在该评论组合（含其下二级回复）的最顶部**
- 作者评论挂"作者"小标签
- 多条作者评论按各自评论热度分内部排序

#### 5.2.2 二级回复

- **固定按时间正序排**（最早的回复在上，最新的在下）——保持对话流的连贯性，符合用户读楼习惯
- **不参与热度排序**：破坏对话顺序反而难读

#### 5.2.3 评论折叠规则

| 规则 | 说明 |
| :-- | :-- |
| 一级评论默认显示 | 前 20 条；点击"加载更多"+20 条 |
| 二级回复默认展开 | 每条一级评论下展开前 2 条；更多用「展开 X 条回复」按钮 |
| 低质评论折叠 | 点赞数 = 0 且发布 > 24 小时的评论，折叠到「查看全部评论」入口里，默认不展示 |
| 违规评论占位 | 被举报删除的评论显示「该评论因违规已被删除」占位文案，不参与排序 |

### 5.3 排序规则的运营干预入口

以下入口由《KPCLIVE 运营后台 PRD》提供，本节仅列举：

| 入口 | 作用 | 对应 PRD |
| :-- | :-- | :-- |
| 置顶动态 | 强制 Feed 流前 3 位，最多 3 条 | 运营后台 §3.1「内容管理」 |
| 下线动态 | 从排序候选池中移除 | 运营后台 §3.1 |
| 加权运营推荐（v1.5 规划） | 个别动态打"运营推荐"标签，给固定 × 1.3 加权 | 待后续版本 |

### 5.4 排序参数配置

> 排序公式中的所有权重数值（点赞 1 / 评论 3 / 转发 5 / 收藏 2、加权倍数 2.0/1.3/1.5/1.5、重算频率 5 分钟、新人保护 100 次曝光、新发布初始分 10 等）均设计为**配置项**，存储于后端配置中心，运营可在不发版的前提下调整。

### 6.5 消息中心（v1.4 整章重构）

> **v1.4 核心变化**：
> ① 从"4 横向 Tab（互动/评论/@我/粉丝）"改为**3 个入口卡片**（评论和@ / 赞和收藏 / 粉丝）——参考小红书/抖音模式（**方案 D**）
> ② 新增**评论被点赞**作为第 8 类消息
> ③ 引入**双层未读语义**——入口卡片"新增"层 + 单条"点击"层（**方案 C**）
> ④ 砍掉 v1.3 中"互动"Tab 的语义歧义（点赞 vs 评论 vs @ 都是互动）
> ⑤ 审核结果**不进**消息中心（由动态详情页顶部黄/红条提示）

#### 6.5.1 功能描述

承载所有他人对当前用户的互动消息（点赞、收藏、转发、评论、回复、@、关注、评论被点赞），用户主动进入「消息」Tab 查看。

> **本期产品形态：纯站内 pull**。不接入任何形式的系统推送（APNs / FCM / TPNS / 极光等），不弹通知中心、不振动、不响铃。用户必须打开 App 才能看到消息。
>
> 此处"系统推送"特指"端外推送通道 + 端内弹窗 banner"，不指消息中心承载的内容性消息（详见 §1.4 措辞澄清）。

#### 6.5.2 入口

| 入口位置 | 表现 |
| :-- | :-- |
| App 底部 Tab Bar「消息」 | 默认入口；任一子类有未读时图标右上角显示**红点**（不显示数字） |
| App 顶部铃铛图标（Feed 流主页右上角） | 备选入口；红点同步亮灭 |

> **双入口红点同步逻辑**：只要 3 个子类（评论和@ / 赞和收藏 / 粉丝）中**任一**有未读，两个入口都显示红点；用户进入消息中心首页后，两处红点**同时**清除（详见 §6.5.8 双层未读语义）。

#### 6.5.3 消息类型清单（8 类）

按"博主作为接收方"梳理：

| # | 类型 | 触发动作 | 文案模板 | 跳转目标 |
| :-- | :-- | :-- | :-- | :-- |
| 1 | 点赞动态 | 他人点赞我的动态 | "{用户 A、B} 等 X 人赞了你的动态" | 该动态详情页 |
| 2 | 收藏动态 | 他人收藏我的动态 | "{用户 A、B} 等 X 人收藏了你的动态" | 该动态详情页 |
| 3 | 转发动态 | 他人转发我的动态 | "{用户 A、B} 等 X 人转发了你的动态" | 该动态详情页 |
| 4 | 评论动态 | 他人在我的动态下发评论 | "{用户名} 评论了你的动态：『……』" | 该评论位置 |
| 5 | 回复评论 | 他人回复我的评论/回复 | "{用户名} 回复了你：『……』" | 该回复位置 |
| 6 | @ 提及 | 他人在动态/评论中 @ 我 | "{用户名} 在动态/评论中提到了你" | 提及所在位置 |
| 7 | 新增关注 | 他人关注了我 | "{用户 A、B} 等 X 人关注了你" | 第一个关注者的个人主页 |
| 8 | **评论被点赞**（v1.4 新增） | 他人点赞我发出的评论/回复 | "{用户名} 赞了你的评论：『……』" | 该评论所在动态详情页 |

**触发边界规则（v1.4 明确）**：

1. **动态作者不收回复链路的非己消息**：A 发动态 → B 评论 A → C 回复 B，A **不**收到通知（A 仅在自己被直接评论或被 @ 时才收 #4 / #6）
2. **二三级以上回复统一归 #5 回复评论**：A 评论 → B 回复 A → C 回复 B，B 收到 #5 "C 回复了你"（不区分层级深度）
3. **@ 提及在动态发布时**：A 发动态时 @ 了 B/C/D，B/C/D **各自独立**收到一条 #6 消息（不聚合）
4. **触发者 = 接收者本人**（自赞自评自@）：**不产生**消息
5. **触发者已被接收者拉黑**：**不产生**消息（详见 §6.9 拉黑规则）

#### 6.5.4 聚合策略

按"价值密度"区分聚合窗口，**聚合在写入消息表时完成**（写时聚合，读时直接拿）：

| 类型 | 聚合 key | 窗口 | 聚合表现 |
| :-- | :-- | :-- | :-- |
| 点赞动态 | `like:{feed_id}:{receiver_id}` | **5 分钟** | "A、B 等 8 人赞了你的动态" |
| 收藏动态 | `favorite:{feed_id}:{receiver_id}` | **5 分钟** | "A、B 等 3 人收藏了你的动态" |
| 转发动态 | `repost:{feed_id}:{receiver_id}` | **1 分钟** | "A、B 等 N 人转发了你的动态" |
| 评论动态 | 不聚合 | — | 每条独立 |
| 回复评论 | 不聚合 | — | 每条独立 |
| @ 提及 | 不聚合 | — | 每条独立 |
| 新增关注 | `follow:{receiver_id}` | **5 分钟** | "A、B 等 3 人关注了你" |
| **评论被点赞**（v1.4） | `like_comment:{comment_id}:{receiver_id}` | **5 分钟** | "A、B 等 N 人赞了你的评论" |

**聚合规则细节**：

1. **窗口内未读时合并**：同一聚合 key 在窗口存活期内、且该聚合消息**未被用户点击**时，新触发合并进同一条（append 到 `actor_list` 前部，更新 `updated_at` 为最新一次）
2. **窗口超时封存**：窗口 TTL 到期后聚合条目"封存"，下次同 key 触发新开一条
3. **已读后新开**：用户**点击**该条聚合消息后即视为已读，**即使在窗口期内**再有同 key 触发，也新开一条
4. **头像展示上限**：列表中展示头像最多 3 个（堆叠展示），剩余以"等 X 人"表示；`actor_list` 后端保留前 50 个用户（**仅供后端审计/扩展使用**，本期前端不开放查看完整列表）
5. **点击单个头像**：跳转该用户个人主页（头像堆叠本身不可点击展开列表，简化交互）
6. **取消互动**（如点赞后取消）：**已生成的消息不撤回**——actor_list 不变，actor_count 不变（已发出的通知不回滚）

**粉丝子页特殊交互**：单条新粉丝消息**右侧带"回关 / 已关注"快捷按钮**（聚合多人的不带；想关注其中某人需点击对方头像进主页）。

#### 6.5.7 消息列表项规则

| 字段 | 说明 |
| :-- | :-- |

| 主文案 | "{用户名} {动作}"（聚合时附带"等 X 人"） |

| 时间 | 遵循 §4.3 时间显示规则；**v1.4 明确**：仅显示"X 分钟前"，**不再显示**"5 分钟窗口聚合"等技术细节文案 |

**未读项视觉**：浅黄底色（#FEFCE8） + 右侧蓝点（#3B82F6）。

#### 6.5.8 已读 / 未读规则（双层未读语义 · v1.4 新增）

v1.4 引入**双层未读语义**（方案 C）：

| 层级 | 概念 | 何时清除 |
| :-- | :-- | :-- |
| **入口卡片层** | 自上次进入主页以来的新增消息数 | 用户进入**消息中心首页**时归零（Tab Bar 红点和铃铛红点也同步消失） |
| **单条消息层** | 某条消息是否被用户**点击过** | 仅当用户：① 点击该单条消息 ② 点击"全部已读"按钮 ③ 长按 → 删除 |

**含义**：
- 用户进入主页 → 入口卡片红色数字徽标立即消失，但**子页内单条仍保持未读样式**
- 用户进子页扫一眼可见哪些是没看过的（黄底+蓝点），可选择"留着以后回复"
- 必须主动**点击单条**或**全部已读**才能消除单条层的未读状态

**触发动作详表**：

| 触发动作 | 已读范围 |
| :-- | :-- |
| 用户进入消息中心**首页** | 入口卡片层归零（3 类卡片徽标全消失 + Tab Bar/铃铛红点同步消失）；单条层**不变** |
| 用户进入某个**子页** | 不再触发任何"已读"——因为入口卡片层已在进首页时归零 |
| 用户**点击单条消息** | 仅该条消息从单条层标记已读（黄底+蓝点消失） |
| 用户点击**首页"全部已读"按钮** | **清除主页层 + 子页单条层**——3 类入口卡片归零 + 所有子页内所有单条标记已读；配 5 秒撤销 toast |
| 用户长按**删除**单条 | 该条直接删除（无论读未读）；配 5 秒撤销 toast |

> **v1.4 修订**：原 v1.3 设计的"子页'全部已读'按钮"已**砍掉**——主页一键全清足够覆盖；子页只保留单条点击与长按删除两种操作，更简约。

**跳转失效消息的已读处理**：消息点击后若关联内容已删除/被审核拦截/触发者已封禁，**仍标记该条为已读** + toast "无法跳转（动态已删除 / 该用户已注销 / ...）"（避免用户清不掉这种"僵尸消息"）。

#### 6.5.9 全部已读 / 删除消息的撤销机制（v1.4 新增）

| 动作 | Toast 表现 |
| :-- | :-- |
| 首页"全部已读"按钮 | 底部 toast："已标记 X 类消息全部已读 ｜ 撤销"，5 秒内点"撤销"恢复 |
| 长按单条删除 | 底部 toast："已删除 1 条消息 ｜ 撤销" |

> 撤销 toast 防止用户误操作丢失消息。

#### 6.5.10 消息存储与清理

| 项 | 规则 |
| :-- | :-- |
| 存储时长 | 默认保留 **90 天**，超过自动清理 |
| 单子页加载策略 | 默认加载 30 条；上滑加载更多，每次 +30 |
| 删除消息 | 长按 → 删除 + 5 秒撤销；**不提供批量删除**（避免误删） |
| 关联动态被删除 | 该消息显示"该动态已被删除"占位，不可跳转；点击仍标记已读 |
| 关联动态被审核拦截 | 同上 |
| 触发者账号注销 / 封禁 | 显示"该用户已注销"，不可跳转其个人主页 |
| 触发者已被本人拉黑 | 拉黑后**不再产生**该用户的互动消息（已存在的不删，但显示为"已屏蔽" + 折叠到底部） |
| 触发者 = 本人 | **不产生**消息 |

#### 6.5.11 异常与边界

| 场景 | 处理 |
| :-- | :-- |
| 消息服务异常 | 整页占位"消息加载失败，请稍后重试" + 重试按钮 |
| **主页 3 类全无消息** | 主页中心占位图 + 文案"暂无消息，去广场看看吧" + CTA"逛逛广场" |
| **子页无消息** | 各自占位文案："暂无评论或@" / "暂无新点赞" / "暂无新粉丝" |
| 入口卡片"全已读时"显示 | 仍展示最新一条预览，**红色数字徽标隐藏**（视觉变安静而非消失） |
| 红点状态不同步 | 后端兜底：每次**进入消息中心首页**时拉取最新未读数，避免红点常驻；不做后台轮询（符合 pull-only 大方向） |
| 用户连续高频触发（疑似刷量） | 写消息时由后端做行为风控：超过频次阈值的触发者，其触发的消息**不进入接收方消息列表**（但互动数据仍计入） |

#### 6.5.12 技术架构（极简）

```
互动事件（点赞 / 评论 / ...）
        │
        ▼
   写入互动表
        │
        ▼
   触发消息聚合（Redis）
   ┌──────────────────────────────────┐
   │ key: notify:{type}:{target_id}:{receiver_id} │
   │ - 命中且未过期 + 该条未读：append actor 到 list│
   │ - 未命中 / 已读：新建 key，TTL = 聚合窗口      │
   └──────────────────────────────────┘
        │
        ▼
   定时器到期 / 阈值触发
        │
        ▼
   ┌──────────────────────────┐
   │ 落库 notification 表       │
   │ 更新 receiver 的未读计数   │
   │   - 入口卡片层 count       │
   │   - 单条层 is_read         │
   └──────────────────────────┘
        │
        ▼
   用户进入消息中心首页 → 拉取消息列表（pull-only）
   用户进入首页 → 入口卡片层 count 归零（不影响单条 is_read）
```

> 全链路无任何第三方推送 SDK，无 APNs / FCM / 厂商通道集成，工程实现轻量。

### 6.6 搜索功能（v1.4 新增 · 极简版）

#### 6.6.1 功能描述

提供站内全局搜索能力，覆盖 **动态 / 用户 / 赛事** 3 类对象。本期采取**极简策略**：无联想词、无历史搜索、无热门搜索、无综合 Tab、无位置/话题独立 Tab。

> 设计原则：保证可用即可，复杂搜索体验留待 v1.5+。

#### 6.6.2 搜索范围

| Tab | 索引字段 | 备注 |
| :-- | :-- | :-- |
| **动态**（默认 Tab） | content / topic_tags / match_tag / 位置 | 标签与文本混合搜索，#手牌分析 自然命中带该标签的动态 |
| **用户** | nickname | 仅按昵称匹配 |
| **赛事** | match_name | 仅按赛事名匹配 |

> 话题、位置不单独做 Tab，已被"动态"Tab 覆盖。

#### 6.6.3 页面状态

**仅 2 个状态**：搜索默认页（点击搜索框进入） + 搜索结果页（提交查询后）。**不做**联想中间态。

##### 搜索默认页

```
┌──────────────────────────┐
│ [🔍 搜索动态/用户/赛事][取消]│  ← 自动聚焦输入框
├──────────────────────────┤
│                           │
│        （空白）            │
│                           │
└──────────────────────────┘
```

不展示历史搜索、热门搜索、推荐入口。

##### 搜索结果页

```
┌──────────────────────────┐
│ [{关键词}_____]    [取消]  │
├──────────────────────────┤
│ [动态] [用户] [赛事]       │  ← 3 Tab，默认动态
├──────────────────────────┤
│ {结果列表（流式）}         │
└──────────────────────────┘
```

#### 6.6.4 排序规则

| Tab | 排序 |
| :-- | :-- |
| 动态 | 命中度 × 发布时间倒序（不复用 §5 热度公式，最简单 ORDER BY） |
| 用户 | 完全匹配 > 前缀匹配 > 模糊匹配；同档按粉丝数倒序 |
| 赛事 | 状态优先（ONGOING > UPCOMING > FINISHED），同档按时间倒序 |

#### 6.6.5 空结果

占位图 + 文案"没有找到相关结果"。**无 CTA**（不做"发布动态"导流，避免低质内容流入）。

#### 6.6.6 合规规则

| 规则 | 说明 |
| :-- | :-- |
| 敏感词搜索 | 关键词命中敏感词库（含 Gamble 等业务红线类别）→ **返回空结果，不报错、不提示**（避免暴露敏感词存在） |
| 触发风控 | 用户短时间多次搜索敏感词 → 触发后端风控审计 |
| 内容过滤 | 仅展示 `audit.status = PASS` 的动态；Review / Block 状态不出现 |
| 被拉黑用户 | 不出现在搜索结果中（参见 §6.9 拉黑双向规则） |

#### 6.6.7 技术实现

| 项 | 选型 | 备注 |
| :-- | :-- | :-- |
| 全文索引 | **MySQL FULLTEXT** | 起步够用，不引 ES |
| 历史/热门搜索 | 不存 | 不实现 |
| 缓存 | 无 | 直接打 DB |

> 若 v1.5+ 业务量提升，可升级为 ElasticSearch。

#### 6.6.8 性能指标

| 接口 | 目标 |
| :-- | :-- |
| 搜索结果首屏 | P95 ≤ 1 秒 |

#### 6.6.9 数据对象

不新增对象，直接复用 `FeedItem` / `UserBrief` / `MatchTag`。

#### 6.9.4 拉黑严格双向规则（v1.4 关键）

拉黑生效后，**双向**应用以下规则：

| 维度 | 规则 |
| :-- | :-- |
| **关注关系** | 立即**双向自动解除**（不论之前是单向关注还是互关） |

| **搜索可见性** | 双向**搜不到**对方 |
| **消息中心** | 拉黑后不再产生该用户的互动消息（已存在的不删，但显示为"已屏蔽" + 折叠到子页底部） |

### 6.10 举报功能（v1.4 修订重写 · 完整闭环）

#### 6.10.1 功能描述

提供用户举报违规内容/账号的能力，对接后端人审队列与超管处理。本期实现**完整闭环**：从举报提交 → 可见性立即变化 → 人审 → 处理结果（内容下线 / 账号处罚 / 驳回）→ 被举报者感知。**举报者本期不感知处理结果**（推迟到 v1.5+）。

#### 6.10.2 举报对象与举报人资格

| 维度 | 规则 |
| :-- | :-- |
| 谁能举报 | 必须**已登录**用户；未登录点举报 → 触发 §4.1 登录引导弹窗 |
| 能举报什么 | 动态（Feed）/ 评论（Comment）/ 用户（User） |
| 自己 → 自己 | 不能举报本人；前端"..."菜单不显示"举报"入口 |
| 举报已拉黑用户 | 允许（拉黑与举报是两件事） |
| 举报蓝 V 官方账号 | **允许**——但 Report 进入人审队列后会标记为"高敏感对象"，由超管亲自处理 |
| 举报金 V 选手 | 允许，与普通用户一致流程 |
| 举报对象已删除 | toast"该内容已不存在" + 关闭 Action Sheet |

#### 6.10.3 举报入口

| 入口位置 | 出处 |
| :-- | :-- |
| Feed 卡片"…" 操作菜单"举报" | §6.1.6 |
| 详情页右上"..." 操作菜单"举报" | §6.2.5 |
| 评论长按菜单"举报" | §6.2.3 |
| 用户个人主页"..." 操作菜单"举报用户" | §6.4.3（他人主页） |

#### 6.10.5 内容可见性策略（C + D 组合 · v1.4 修订核心）

提交举报后，根据**举报原因严重程度**采取不同可见性策略：

| 举报原因 | 提交瞬间的可见性 | 后端动作 |
| :-- | :-- | :-- |
| **涉黄 / 涉政 / 涉博彩**（高敏感三类） | **立即对所有人下线**（标记"审核中"）| 进入**高优先级人审队列**，30 分钟内处理 |
| **谩骂攻击** | **仅举报者前端过滤**（其他用户照常可见） | 进入**中优先级队列**，2 小时内处理 |
| **广告灌水 / 其他** | 仅举报者前端过滤 | 进入**低优先级队列**，24 小时内处理 |

**"仅举报者过滤"的实现**：在 Feed/详情/评论/用户主页拉取时，过滤掉"我举报过且 status = PENDING/REJECTED 的对象"。

**"立即下线"的视觉表现**：
- 动态被立即下线时，对所有非作者用户隐藏（Feed 流不出现、详情页 404）
- 对作者本人：详情页顶部显示**橙色提示条**"⏳ 你的动态正在审核中，他人暂不可见"（与 §4.2.4 Review 黄条一致）

#### 6.10.6 人审处理（4 选 1 判定）

人审员在运营后台（详见《KPCLIVE 运营后台 PRD v1.1 §4 举报队列处理》）逐条处理 Report：

```
人审员打开一条 Report
   ↓
看到：举报原因 + 补充说明 + 关联内容 + 该对象已累计被举报次数 + 举报者历史画像
   ↓
做 4 选 1 判定：
   ┌──────────────────────────────────────────┐
   │ A. 判定违规 → 内容下线                     │
   │    Report.status = HANDLED                │
   │    被举报内容置为 BLOCK 状态（同 §4.2.4）  │
   │    被举报内容详情页显示红条                 │
   │                                            │
   │ B. 判定严重违规 → 内容下线 + 账号处罚       │
   │    Report.status = HANDLED                │
   │    走 §4.2.7 账号风险流程：                │
   │      · 第 1 次警告弹窗                     │
   │      · 第 2 次发布权限冻结 7 天             │
   │      · 第 3 次永久封号                     │
   │                                            │
   │ C. 判定不违规 → 驳回                       │
   │    Report.status = REJECTED               │
   │    如内容曾被立即下线（涉黄/政/赌）→ 恢复可见│
   │    举报者侧保留"仅举报者过滤"（尊重意愿）   │
   │                                            │
   │ D. 重复 / 误报 → 自动跳过                  │
   │    Report.status = DUPLICATE              │
   └──────────────────────────────────────────┘
```

**处理时效（SLA）**：见 §6.10.5 表中"后端动作"列。

#### 6.10.7 超管（超级管理员）能力（v1.4 修订新增）

超管不走"举报 → 队列 → 人审"流程，可**直接处理**：

| 超管能力 | 用途 | 详细规则 |
| :-- | :-- | :-- |
| **一键删帖** | 立即下线任何动态 | 跳过队列，直接置 BLOCK 状态；详情页红条文案为**"该内容已被管理员下线"**（区别于"审核未通过"） |
| **一键删评论** | 立即下线任何评论 | 同上 |
| **一键封号** | 立即冻结/封禁账号 | 跳过 §4.2.7 警告流程，直接到对应处罚级别 |
| **代发举报** | 运营主动发现违规 | 把违规对象推入人审队列（不直接下线） |
| **解封 / 撤销下线** | 误判回滚 | 恢复内容可见 / 恢复账号 |
| **审核词库回灌** | 把通过的违规特征加机审 | 关键词 / 图片 MD5 加进腾讯云自定义词库 |

> 上述超管功能的页面、权限、操作流程详见**《KPCLIVE 运营后台 PRD v1.1》§4 内容治理与超管能力**。本 PRD 仅引用。

**超管删除 vs 普通审核未通过的视觉区分**：

| 下线原因 | 详情页红条文案 |
| :-- | :-- |
| 机审 Block（§4.2.4） | "⛔ 审核未通过：{违规类别}" |
| 举报 → 人审判定违规 | "⛔ 内容因违规被下线" |
| **超管一键删** | **"⛔ 该内容已被管理员下线"** |

#### 6.10.8 被举报者感知

| 判定结果 | 被举报者看到什么 |
| :-- | :-- |
| 立即下线（涉黄/政/赌 1 次举报即触发） | 详情页橙色条"⏳ 你的动态正在审核中，他人暂不可见" |
| 人审 A. 内容下线 | 详情页红条"⛔ 内容因违规被下线" |
| 人审 B. 账号被罚 | 走 §4.2.7 流程：第 1 次警告弹窗 → 冻结 → 封号 |
| 人审 C. 驳回（误举报） | 内容恢复可见，**被举报者完全无感知** |
| 超管一键删 | 详情页红条"⛔ 该内容已被管理员下线" |

#### 6.10.9 举报者感知（本期最简）

| 时机 | 举报者感知 |
| :-- | :-- |
| 提交后 | toast"举报已收到，我们会尽快处理" |
| 提交后到人审完成期间 | 被举报对象按 §6.10.5 策略对该举报者前端过滤 |
| 人审完成后 | **本期无任何通知**（推迟到 v1.5+） |

#### 6.10.10 防滥用机制（v1.4 修订加强）

| 滥用类型 | 防护机制 |
| :-- | :-- |
| 同一人重复举报同一对象 | 24h 内只接受 1 条 → toast"你已举报过该内容，无需重复举报" |
| 同一人短时间海量举报 | ≥20 条 / 1 小时 → 触发风控审计（暂停其举报能力 24h） |
| 长期通过率低的举报者 | **本期写规则但不启用**——降权逻辑需累计至少 3 个月数据后方有意义，**v1.5 启用**。届时实现：通过率 < 30% 自动降权（队列优先级下移 + 立即下线效果取消） |
| 长期被举报通过的用户 | 自动标记"风险账号"，后续机审策略加严 |

#### 6.10.11 与机审（§4.2）的关系

**互补**：
- 机审 = **发布时主动检查**（覆盖率高、召回率有限）
- 举报 = **用户被动发现后上报**（兜底机审漏召的违规，如图文混合、表情包、谐音字）

举报通过的违规特征（关键词、图片 hash）→ 由超管手动**回灌到腾讯云 TMS / IMS 自定义词库**，逐步提升机审准确率（操作详见运营后台 PRD）。

#### 6.10.12 异常与边界

| 场景 | 处理 |
| :-- | :-- |
| 网络异常 | toast"提交失败，请稍后重试"，Action Sheet 保留输入内容 |
| 重复举报（24h 内） | toast"你已举报过该内容，无需重复举报"，关闭 Action Sheet |
| 关联内容已删除 | toast"该内容已不存在"，关闭 Action Sheet |
| 举报对象 = 本人 | 入口不展示（前端拦截） |
| 举报对象已被本人拉黑 | 入口正常显示（拉黑后仍可举报） |
| 举报对象为蓝 V 官方账号 | 入口正常显示；Report 标记 high_sensitive=true，路由给超管亲自处理 |
| 举报立即下线触发风控（举报者被降权） | 该举报"仅举报者过滤"，不再触发"立即下线"；其他用户照常可见，等人审 |

### 7.8.1 入口卡片层未读计数（v1.4 新增 · 实现建议）

为支撑方案 C 双层未读语义，"入口卡片层"独立于单条 `is_read`。实现方式由研发选择（PRD 不强制）：

**实现方案 A（独立计数表，推荐高并发场景）**

| 字段名 | 类型 | 说明 |
| :-- | :-- | :-- |
| user_id | String | 用户 ID |
| subpage | Enum | COMMENT_AND_AT / LIKE_AND_FAVORITE / FAN |
| unread_count | Int | 入口卡片层未读数（自上次进入消息中心首页起的新增数） |
| last_visited_at | Timestamp | 用户上次进入消息中心首页的时间 |
| latest_notification_id | String | 该 subpage 下最新一条消息 ID（用于入口卡片预览） |

**实现方案 B（复用 notification 表 + 索引，推荐低流量起步）**

直接 SQL：`SELECT COUNT(*) FROM notification WHERE receiver_id = ? AND subpage = ? AND created_at > last_visited_at`，配合 `(receiver_id, subpage, created_at)` 复合索引。`last_visited_at` 单独存到 user_meta 表。

**两方案的"行为契约"必须一致**（产品层）：
- 每次产生新 notification（非自赞自评、非被拉黑）→ 入口卡片层"新增 +1"
- 用户进入消息中心**首页** → 3 个 subpage 的入口卡片层全部归零；记录 `last_visited_at`
- 进入子页**不**清零
- "全部已读"按钮 → 既清入口卡片层（归零）又清单条层（is_read = true）

## 附录 D：排序规则伪 SQL（v1.2 新增）

```sql
-- 步骤 1：批量计算近 7 天动态的基础热度分（每 5 分钟一次定时任务）
UPDATE feed_items f
SET ranking.hot_score = (
        COALESCE(stats.like_count,0)     * @weight_like
      + COALESCE(stats.comment_count,0)  * @weight_comment
      + COALESCE(stats.repost_count,0)   * @weight_repost
      + COALESCE(stats.favorite_count,0) * @weight_favorite
    ) / (TIMESTAMPDIFF(HOUR, created_at, NOW()) + @time_decay_base),
    ranking.last_computed_at = NOW()
WHERE created_at >= NOW() - INTERVAL @content_ttl_days DAY
  AND audit.status = 'PASS';

-- 步骤 2：批量计算加权后的最终排序分
UPDATE feed_items f
LEFT JOIN users u ON f.author_id = u.user_id
SET ranking.weighted_score = ranking.hot_score
    * CASE u.verify_type WHEN 'BLUE_V' THEN 2.0 WHEN 'GOLD_V' THEN 1.3 ELSE 1.0 END
    * CASE WHEN f.tags.match_tag.status = 'ONGOING' THEN 1.5 ELSE 1.0 END
WHERE ranking.last_computed_at = NOW();
-- 注：「关注用户 × 1.5」依赖当前用户上下文，无法在批量任务中算，在用户拉 Feed 时即时叠加

-- 步骤 3：用户拉 Feed 时按 weighted_score 倒序取数，先取置顶 3 条，再取剩余按分排序
SELECT * FROM feed_items
WHERE audit.status = 'PASS'
  AND created_at >= NOW() - INTERVAL @content_ttl_days DAY
ORDER BY
    ranking.is_pinned DESC,
    ranking.pin_order ASC,
    (ranking.weighted_score
      * CASE WHEN author_id IN (SELECT following_id FROM user_follows WHERE user_id = ?) THEN 1.5 ELSE 1.0 END
    ) DESC
LIMIT 20 OFFSET ?;
```

> 实际工程实现可结合 Redis ZSet 缓存 weighted_score，5 分钟刷新一次；用户拉 Feed 时基于 ZSet 分页+关注关系即时加权。
